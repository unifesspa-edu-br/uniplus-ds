Críticos (P0) — bloqueiam produção

Seis defeitos materiais que afetam usuários reais hoje. Quatro deles são consequência da mesma raiz: a paleta de neutros não tem variantes para o tema escuro, então qualquer componente que referencia --color-neutral-* diretamente herda valores claros no dark.

P0
F-001 · base.css

Ring de foco sobrescreve o border-radius do elemento

Onde
base.css : 55–59

:where(a, button, [role="button"], input, select, textarea, summary, [tabindex]):focus-visible {
  outline: var(--focus-ring-width) solid var(--focus-ring-color);
  outline-offset: var(--focus-ring-offset);
  border-radius: var(--radius-xs); ← problema
}
Impacto

Todo botão .btn (raio pill 9999px), avatar (raio 50%), uploader (radius-lg), back-to-top (circle) achata para 2 px no momento em que recebe foco por teclado. Visual confuso, parece bug, e usuários com cognição reduzida podem interpretar como erro do sistema.

Recomendação

Remover a declaração border-radius da regra :focus-visible. O outline dos browsers modernos já segue o raio nativo do elemento — não é preciso forçar nada. Se a intenção era padronizar o ring em elementos sem raio explícito, fazer isso apenas em um conjunto restrito (input, textarea, summary) e não em button / [role="button"].

P0
F-002 · components.css

Botão :disabled ilegível no tema escuro

Onde
components.css : ~58 (.btn[disabled], .btn:disabled)

background: var(--color-neutral-200); /* #e2e8f0 */
color: var(--color-neutral-500); /* #8899a8 */
Impacto

Os tokens --color-neutral-* não são redefinidos no bloco [data-theme="dark"]. No escuro o botão desabilitado renderiza fundo cinza-claro #e2e8f0 com texto cinza-azulado #8899a8 — contraste ~1.6 : 1, falha WCAG 1.4.3 e 1.4.11. Idêntico padrão afeta .btn--tertiary --btn-bg-active.

Recomendação

Criar tokens semânticos de "disabled" que flutuem por tema: --surface-disabled + --text-disabled. No light apontam para neutral-200/500; no dark, para slate-700/400. Substituir todas as referências diretas a --color-neutral-* nos componentes por tokens semânticos. Aplica-se também a .input:disabled, .pager__btn:disabled, etc.

F-003 · components.css

Skeleton loader invisível no tema escuro

Onde
components.css : ~580 (.skeleton)

background: linear-gradient(90deg,
  var(--color-neutral-100) 0%,
  var(--color-neutral-200) 50%,
  var(--color-neutral-100) 100%);
Impacto

Mesma raiz do F-002. No dark o skeleton vira um bloco claro #f1f5f9 → #e2e8f0 sobre fundo #1e293b — contraste tão alto que parece conteúdo final, não placeholder. Quebra a percepção de "carregando" justamente no momento mais sensível da UX.

Recomendação

Introduzir --skeleton-from e --skeleton-to como tokens semânticos. Light: neutral-100/200. Dark: slate-800/700 (sutilmente mais claro que --surface-card). Bonus: usar color-mix(in oklab, var(--surface-card) 92%, var(--text-primary)) para derivar automaticamente do par superfície/texto vigente.

F-004 · components.css

Avatar com iniciais ilegível no tema escuro

Onde
components.css (.avatar, .user-chip__avatar, .topbar__mark hover)

background: var(--color-primary-100); /* dark: #1e3a5f */
color: var(--color-primary-800); /* dark: #2563eb */
Impacto

No dark, primary-100 vira azul-escuro #1e3a5f e primary-800 vira azul-vivo #2563eb. Iniciais "JF" em azul-vivo sobre azul-escuro: contraste estimado ~1.8 : 1, falha AA. Atinge avatar do user-chip, sidebar e qualquer card com autor.

Recomendação

Tokenizar --avatar-bg e --avatar-fg. No light: primary-100 + primary-800 (como hoje). No dark: color-mix(in srgb, var(--color-primary) 25%, var(--surface-card)) + #fff. Alternativa: aplicar opacidade 0.15 ao primary atual como fundo e usar branco como texto — funciona nos dois temas com uma única regra.

F-005 · components.css

Stepper "is-done" falha contraste no escuro

Onde
components.css (.stepper__dot.is-done)

background: var(--color-success-600); /* dark: #4ade80 (verde-claro) */
color: #fff;
Impacto

O token --color-success-600 foi redefinido para verde-claro no dark (para servir como cor de texto sobre fundo escuro). Aqui ele é usado como fundo, com texto branco — contraste ~2.1 : 1. Falha AA. O mesmo efeito atinge a linha conectora ::before do timeline.

Recomendação

Trocar --color-success-600 por --color-success-solid aqui (já existe como token "fica-escuro-em-ambos-os-temas", justamente para texto branco sobre fundo colorido). A mesma correção aplica-se a .toast-region e qualquer fundo de feedback que carregue texto branco.

F-006 · tokens.css

--border-strong falha 1.4.11 no tema light

Onde
tokens.css : ~75 (--border-strong: var(--color-neutral-400) /* #94a3b8 */)
contraste vs #ffffff = 2.7 : 1 · WCAG 1.4.11 exige ≥ 3.0 : 1
Impacto

Já registrado em docs/contrast.md como pendente. Aplica-se em borda hover de input e bordas de form preenchido. Em telas claras, usuários com baixa visão não percebem o estado hover do campo. Bloqueia auditoria final do CTIC.

Recomendação

Mudar --border-strong de #94a3b8 para #64748b (slate-600) no light. Novo contraste vs branco: 4.4 : 1. Atualizar a matriz em docs/contrast.md e marcar o item pendente como concluído. Verificar visualmente que o hover do input continua percebível e distinto de focus-ring.

Tokens — paleta, tipografia, espaçamento

O arquivo está bem estruturado e generoso. Os problemas estão na semântica: tokens com nomes diferentes apontando para o mesmo valor (drift na hora de evoluir), neutros sem track escuro, e o bloco do tema escuro repetido literalmente duas vezes.

P1
F-007 · tokens.css

Bloco do dark theme duplicado byte-a-byte

Onde
tokens.css : ~242 (:root[data-theme="dark"])
tokens.css : ~290 (@media prefers-color-scheme: dark + [data-theme="auto"])
Impacto

São ~50 linhas idênticas em dois lugares. Toda mudança no escuro precisa ser feita em duplicata. Risco alto de divergência silenciosa entre o que o usuário vê quando escolhe "escuro" manualmente vs. quando o sistema dele está no escuro.

Recomendação

Extrair o corpo do escuro para um placeholder via @layer ou usar o pattern de seletor combinado: :root[data-theme="dark"], :root[data-theme="auto"]:where(...) dentro de uma única @media. Solução mais simples: definir os tokens em uma classe utilitária .theme-dark e aplicar via dois seletores na mesma regra.

P1
F-008 · tokens.css

--surface-banner colapsa em --surface-card no dark

Onde
tokens.css (dark) : --surface-banner: #1e293b;
tokens.css (dark) : --surface-card: #1e293b;
Impacto

O comentário no token explica corretamente que "banner em dark NÃO pode usar azul-marca". Mas colapsar com card destrói a hierarquia visual: o carrossel de editais em destaque, no dark, fica indistinguível dos cards abaixo dele. Toda a intenção do hero some.

Recomendação

Usar color-mix(in srgb, var(--color-primary) 12%, var(--surface-card)) — gera um #1e293b levemente azulado, visualmente distinto do card sem cair em "azul-on-azul". Ou um slate ligeiramente mais escuro (#172033) para diferenciar como "fundo mais profundo".

P1
F-009 · tokens.css

Três tokens com o mesmo valor e semânticas diferentes

Onde
--color-primary-800 = #0c326f
--text-heading = var(--color-primary-800)
--color-info-700 = #0c326f (mesmo hex literal)
Impacto

Hoje os três casam, mas se amanhã alguém quiser escurecer só o heading (ou só o info), terá que descobrir que dois tokens compartilhavam o valor literal — não a derivação. Drift por descuido é o defeito mais comum em DS maduros.

Recomendação

Reescrever --color-info-700 como var(--color-primary-800) em vez de hex literal. Mesmo princípio para --text-link e --color-info-600 no dark (ambos #93c5fd): apontar um para o outro, não duplicar. Adicionar uma regra no Stylelint que impeça hex repetido entre tokens.

P1
F-010 · tokens.css

--text-muted falha AA quando usado em texto normal

Onde
--text-muted = #94a3b8 → 3.4 : 1 vs #fff
usado em: .field__hint, .card__meta, .uni-uploader__item-meta, .pager (caption)
Impacto

3.4:1 só passa AA-Large (≥18px regular ou ≥14px bold). O hint do form e o meta do card são texto normal de 12 px — falham 1.4.3.

Recomendação

Subir --text-muted para #64748b (slate-600 — 4.7:1 vs branco). Onde o efeito "muito sutil" for desejado (placeholder de input, p.ex.), criar um --text-placeholder separado e documentar explicitamente que só serve para placeholder (que é UI, não texto-conteúdo, e tem 3:1 como floor).

P2
F-011 · tokens.css

Inter carregado da CDN do Google com pesos sobrando

Onde
tokens.css : 23
@import url("https://fonts.googleapis.com/css2?family=Inter:ital,wght@0,400;0,500;0,600;0,700;0,800;1,400;1,600&family=JetBrains+Mono:wght@400;600&display=swap");
Impacto

Dois problemas. (1) Privacidade — para um produto federal sob LGPD, depender da CDN do Google envia IP do candidato ao Google em toda navegação. (2) Bandwidth — está carregando peso 800 e itálicos que o DS declara não usar (apenas 400/500/600/700, sem itálico).

Recomendação

Self-host: baixar Inter (axis variable wght 400–700) e JetBrains Mono (400/600), servir via libs/shared-ui/src/fonts/ com @font-face font-display: swap. Reduzir o import a 4 pesos. Documentar no docs/contributing.md que adicionar peso novo passa por ADR de bundle-size.

F-013 · tokens.css

Sem token para "superfície elevada"

Onde
--surface-card é usado para card, modal, drawer, toast e popover indiscriminadamente.
Impacto

No light não dá pra distinguir um popover sobre um card. No dark com --shadow-3: rgba(0,0,0,.7) a sombra somem em fundo escuro, então a sobreposição depende exclusivamente de um border-subtle de 1px.

Recomendação

Introduzir --surface-elevated (light: #ffffff + shadow; dark: #28344a, ligeiramente mais claro que --surface-card) e usar em modal, drawer, dropdown, toast, tooltip.

F-014 · tokens.css

--shadow-* invisível no escuro

Onde
dark: --shadow-3 = 0 12px 32px rgba(0,0,0,.7)
Impacto

Sombra preta sobre fundo quase-preto não é percebida. Modal e drawer no dark perdem profundidade.

Combinar sombra preta com um glow sutil escurecedor da borda: 0 12px 32px rgba(0,0,0,.7), 0 0 0 1px rgba(0,0,0,.4). Ou subir a luminância do --surface-elevated (F-013) para criar contraste por luz, não por sombra — convenção comum em Material You / iOS dark.

F-025 · components.css

Carrossel oculta slides via height: 0

Onde
.carousel__slide[aria-hidden="true"] { visibility: hidden; height: 0; padding: 0; }
Impacto

Os slides são filhos do .carousel__track que usa transform: translateX() para navegar. Colapsar slide para height:0 muda a largura do track e quebra o cálculo do translate. O componente fica entre dois padrões (transform-slider vs. show-one-hide-rest) e nenhum funciona bem.

color-scheme: light redundante no [data-theme="dark"]

Onde
tokens.css : 233 — :root[data-theme="dark"], :root[data-theme="auto"] { color-scheme: light; }
tokens.css : 237 — :root[data-theme="dark"] { color-scheme: dark; }
Impacto

A primeira regra inclui [data-theme="dark"] e seta color-scheme: light, depois a segunda regra sobrescreve para dark. Funciona, mas é confuso ler.

Recomendação

Limpar: :root[data-theme="auto"] { color-scheme: light; } e :root[data-theme="dark"] { color-scheme: dark; } separados, sem grupo.

Tema alto contraste

Este é o tema mais frágil hoje. Cobre texto base, mas qualquer componente que use cores de feedback (alert, tag, timeline) ainda renderiza com paleta light/dark — quebrando o propósito do modo.

P1
F-035 · tokens.css

Feedback (success/warning/danger/info) não definidos no contrast

Onde
:root[data-theme="contrast"] { ... }
— redefine surface/text/border/primary/focus
— NÃO redefine --color-success-*, --color-warning-*, etc.
Impacto

Quando Carla (NVDA + zoom 200%, persona oficial) ativa o modo contraste e abre um alert de sucesso, vê texto verde-escuro #0a5a16 sobre fundo verde-claro #e7f4e9 — quebra a regra "fundo preto, texto amarelo" do modo. O alert vira uma ilha de paleta light dentro de uma tela contrast.

Recomendação

Definir todos os tokens de feedback no bloco contrast. Convenção: todos os "50" viram #000, os "600"/"700" viram #ffff00, com border explícito (porque tudo é preto). Aceitar que o modo contrast deliberadamente colapsa as semânticas de cor — a hierarquia agora é tipográfica e por ícone, não cromática.

P2
F-036 · tokens.css

Bordas do contrast todas iguais

Onde
--border-subtle = --border-default = --border-strong = #ffff00
Impacto

O ponto da hierarquia (subtle/default/strong) some. No contrast, todos os elementos têm borda da mesma força — visualmente "tudo grita".

Recomendação

Manter strong = #ffff00 (3px) para foco e estados ativos. Default = #ffff00 (1px). Subtle = #666666 (decorativo) ou usar dashed para distinguir sem cor. Validar com usuários do NAIA antes de fechar.

P3
F-037 · tokens.css

color-scheme errado no contrast

Onde
:root[data-theme="contrast"] { color-scheme: light; }
Impacto

Fundo preto + color-scheme: light faz scrollbars nativas renderizarem claras sobre fundo preto — contraste forte com o resto. Convenção: color-scheme: dark em temas de fundo preto.

Recomendação

Mudar para color-scheme: dark.

UI kits — Portal & Admin

Os dois kits demonstram bem o sistema; defeitos aqui são pequenos mas perceptíveis.

P1
F-038 · ui_kits/portal

Links institucionais escondidos em mobile

Onde
components.css : @media (max-width: 640px) { .gov-bar__links a:not(:first-child) { display: none; } }
Esconde "Acessibilidade" e "Privacidade" em mobile.
Impacto

"Acessibilidade" é direito do usuário e "Privacidade" cobre obrigação LGPD. Esconder em mobile (onde a maioria do tráfego do candidato Unifesspa vai estar) cria fricção legal. Gov.br DS exige presença em todas as breakpoints.

Recomendação

Manter visíveis em mobile. Se faltar espaço, colapsar em um menu "...", não esconder. Alternativa: mover esses dois links para o footer (que sempre é visível) e manter na gov-bar só "Mapa do site".

P2
F-039 · ui_kits/portal

Subnav com scroll horizontal sem indicação visual

Onde
.subnav { overflow-x: auto; scrollbar-width: thin; }
Impacto

Em mobile, links "cortados" no canto direito não sinalizam que existem mais. Usuários não descobrem. Comum em portais com 6+ seções.

Recomendação

Adicionar fade gradient no canto direito quando há mais conteúdo: mask-image: linear-gradient(90deg, #000 0 95%, transparent). Ou setas pequenas que aparecem quando há overflow (mais explícito, melhor para baixa visão).

P2
F-040 · ui_kits/admin

Sidebar tooltip duplica padrão do DS

Onde
ui_kits/admin/index.html :85–110 — bloco CSS dentro de <style> do kit cria seu próprio tooltip da sidebar colapsada.
Impacto

Funciona, mas reimplementa o padrão [data-tooltip] que está em components.css. Quando o componente Angular for criado, vai precisar decidir qual versão herdar.

Recomendação

Resolver F-024 (left/right) primeiro. Aí o sidebar colapsado consome [data-tooltip][data-tooltip-position="right"] nativo.

Docs & preview cards

Doc é uma das partes mais fortes do projeto. Os achados são todos de "doc-out-of-sync" com o código.

P1
F-041 · docs/component-api.md

Skeleton e Spinner descritos como "TODO" mas já existem

Onde
component-api.md : "Skeleton loader — TODO" / "Loading spinner — pendente"
components.css : skeleton (~574-595) e spinner (~601-625) implementados
Impacto

Quem ler a doc não saberá que existe. Vai duplicar.

Recomendação

Mover de "TODO" para a seção de componentes oficiais, com anatomia + ARIA. Adicionar versão Angular standalone como pattern. Atualizar o backlog "O que vem por aí" removendo skeleton e spinner.

P1
F-042 · components.css + docs/cursor-pagination.md

Contrato cursor contradiz entre CSS-comment e doc

Onde
components.css (.pager comment): "API retorna { items, nextCursor, prevCursor, hasNext, hasPrev }"
docs/cursor-pagination.md : "Body é array puro. NÃO existe hasNext no body. Cursores vêm no header Link (RFC 8288)."
Impacto

Doc canônica diz uma coisa, comentário inline do CSS diz outra. Quem chegar pela ordem inversa (lendo components.css primeiro) vai implementar errado e quebrar no QA.

Recomendação

Reescrever o comentário do .pager linkando para docs/cursor-pagination.md e mantendo apenas o que é UI (markup esperado, estados disabled, copy). Não duplicar contrato de API em CSS.

P2
F-043 · preview/colors-themes.html

Paleta do dark no preview difere da produção

Onde
preview/colors-themes.html : usa #0a1628 como fundo do "DARK · EXPLICIT"
tokens.css (dark) : --surface-page = #0f172a
Impacto

O cartão do Design System tab mostra um dark que não bate com o que vai pra produção.

Recomendação

Refatorar o card para consumir tokens em vez de hex inline. Aplica-se a todos os 45 preview cards: idealmente nenhum deveria ter hex hardcoded.

P2
F-044 · docs/tokens.md

Stylelint config sem arquivo correspondente

Onde
tokens.md descreve regras Stylelint detalhadas, mas não há .stylelintrc.* no repo.
Impacto

Regra que não é executada = regra inexistente. Os achados F-020 e F-028 só existiriam se o lint estivesse rodando.

Recomendação

Materializar como .stylelintrc.json no root, rodar como pre-commit hook. Documentar no contributing.md.

P3
F-045 · README.md

@theme inline com self-reference confunde

Onde
@theme inline {
  --color-primary: var(--color-primary);
  ...
}
Impacto

Funciona no Tailwind 4/5 mas a leitura é confusa — parece bug (variável referenciando ela mesma). Quem não conhece o pattern do @theme inline vai tropeçar.

Recomendação

Manter como está (é idiomático em Tailwind 4), mas adicionar comentário inline: /* Tailwind reads the right-hand var() at compile time and inlines its value */.
